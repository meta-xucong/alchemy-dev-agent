# V2.33 Artifact File Previews

## Goal

V2.33 turns delivery evidence files into a reviewable artifact contract.

The system already records screenshots, generated native UI test drafts, static artifact files, and generated CI workflow evidence in `run.json`, `artifact_report`, `delivery_report`, and `delivery_evidence`. The missing layer was a safe API/UI path for a reviewer to open those files without manually browsing the run directory.

## Scope

This phase adds:

- `artifact_manifest` to API delivery responses.
- `GET /projects/{project_id}/runs/{run_id}/artifacts` for a run-scoped manifest.
- `GET /projects/{project_id}/runs/{run_id}/artifacts/{artifact_id}` for file content.
- Browser-console artifact previews for screenshots and openable links for text artifacts.

This phase does not change:

- task planning,
- worker behavior,
- evaluator scoring,
- GitHub flow,
- retry logic,
- native UI test generation behavior.

## Manifest Sources

The manifest is built only from paths already present in run evidence:

- `artifact_report.browser_verification.screenshots`
- `delivery_report.artifact.screenshots`
- `artifact_report.native_ui_tests.files`
- `delivery_report.artifact.native_ui_tests.files`
- `runtime_state.repository.native_ui_tests.files`
- `artifact_report.artifact_files`
- `delivery_report.artifact.artifact_files`
- `generated_ci.workflow_path`
- `generated_ci.evidence`

The API never accepts a caller-provided filesystem path for artifact serving.

## Manifest Shape

```json
{
  "project_id": "proj_example",
  "run_id": "run_001",
  "items": [
    {
      "artifact_id": "screenshot_initial_screenshot_browser_initial_png_abc123",
      "kind": "screenshot",
      "label": "initial screenshot",
      "path": "browser_initial.png",
      "media_type": "image/png",
      "size_bytes": 12345,
      "preview": "image",
      "url": "/projects/proj_example/runs/run_001/artifacts/screenshot_initial_screenshot_browser_initial_png_abc123"
    }
  ]
}
```

Public manifest items never include absolute filesystem paths.

## Safe Serving Rules

An artifact file is served only when all conditions are true:

- The path came from persisted run evidence.
- The resolved file exists.
- The resolved file is inside the project run directory, project storage directory, or the recorded repository/worktree directory.
- The artifact id is generated by the manifest and resolved back through the manifest.

Text-like source artifacts, including HTML, JavaScript, TypeScript, CSS, JSON, Markdown, YAML, and generated test files, are served as `text/plain; charset=utf-8`. This prevents generated HTML artifacts from executing inside the local API origin during review.

Image files keep their image media type so screenshots can render inline.

## UI Behavior

The browser console renders an `Evidence Artifacts` section in the Delivery panel:

- screenshots render as inline thumbnails and link to the served file;
- generated native UI tests open as text previews;
- generated CI workflows open as text previews;
- artifact files open as text previews.

Raw JSON remains available in the Delivery output for audit.

## Acceptance Criteria

- Delivery responses include `artifact_manifest`.
- Run artifact manifests include screenshots, native UI test drafts, static artifact files, and generated CI workflows when those files exist.
- Artifact content routes return the underlying file bytes with safe media types.
- Manifest output does not expose absolute paths.
- Out-of-root reported files are ignored.
- Static console assets include the artifact preview section.
