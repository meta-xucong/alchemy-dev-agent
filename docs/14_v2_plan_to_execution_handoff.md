# V2.5 Plan-To-Execution Handoff Runtime

## Purpose

V2.5 connects document-driven planning to the runtime execution loop.

The implemented flow is:

```text
ProjectBrief
  -> ContextBundle
  -> TaskGraph
  -> RuntimeHandoff
  -> RuntimeState
  -> CodexWorkerInput packages
  -> Orchestrator dry-run execution
  -> Evaluation gate
```

This closes the gap between V2.4 planning and the existing runtime engine.

## Handoff Contract

`RuntimeHandoff` converts:

- `ProjectBrief`
- `ContextBundle`
- generated `TaskGraph`

Into:

- `RuntimeState`
- repository metadata
- blocker carryover
- document-aware done criteria
- executable worker task packages

It does not create a second state model. It uses the existing `RuntimeState`, `TaskGraph`, `TaskNode`, and `CodexWorkerInput` contracts.

## Release Evidence

The existing DONE gate requires GitHub execution evidence. Document-driven task graphs generated by V2.4 end at review, so V2.5 appends a release task during handoff when the graph does not already contain one.

The release task records:

- branch evidence
- commit evidence
- pull request evidence
- CI or dry-run delivery evidence

In dry-run mode, this evidence is deterministic and does not mutate remotes.

## Worker Packages

Each worker package includes:

- task ID
- objective
- agent-specific goal
- task description
- acceptance criteria
- repository path
- branch name
- assigned agent
- upstream task IDs
- relevant files
- verification commands
- constraints

This package is the boundary passed to Codex worker execution.

## Verification

V2.5 is verified by:

- Building a ProjectBrief from a development document.
- Building a ContextBundle with repository evidence.
- Generating a TaskGraph.
- Creating RuntimeState through `RuntimeHandoff`.
- Building Codex worker input packages.
- Running the generated graph through `Orchestrator` dry-run execution.
- Reaching the existing DONE gate with dry-run GitHub evidence.

## Current Limitations

- Real Codex mutation of arbitrary repositories is still gated behind the existing `real_codex` adapter.
- Live GitHub PR and CI evidence are still handled by the existing optional `real_github` adapter, not by a product API.
- V2.38 adds lightweight contradiction warnings before handoff. Deep semantic
  requirement validation remains future work.

Later phases add local API project creation, browser upload, execution
monitoring, real Codex configuration, isolated worktrees, and recovery-run
resume. See `docs/17_v2_local_api_runtime.md`,
`docs/18_v2_browser_ui_async_runtime.md`, `docs/25_real_run_worktree_lifecycle.md`,
and `docs/26_resumable_real_worker_execution.md`.
